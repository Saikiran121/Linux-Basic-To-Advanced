# Linux `journalctl` Command: In-Depth Guide

The `journalctl` command is a powerful utility used to query and read log data collected by `systemd`. In modern Linux distributions (like Ubuntu, CentOS, Fedora), `systemd-journald` is the central logging service that collects logs from the kernel, system services, and applications. 

`journalctl` allows you to filter, format, and manage these logs efficiently.

---

## 1. Basic Usage

*   **View all logs**:
    ```bash
    journalctl
    ```
    *(Opens the entire system log in a pager (usually `less`). You can use arrow keys to navigate and `q` to quit).*

*   **Follow Live Logs (Tail)**:
    Just like `tail -f`, you can watch logs as they are written in real-time.
    ```bash
    journalctl -f
    ```

*   **Reverse Order (Newest First)**:
    By default, `journalctl` shows the oldest logs first. To see the newest logs at the top:
    ```bash
    journalctl -r
    ```

*   **Show only a specific number of lines**:
    ```bash
    journalctl -n 50
    ```
    *(Shows the last 50 lines of the log).*

---

## 2. Filtering by Service (Unit)
When troubleshooting a specific application (like a web server or database), you usually only want to see its logs.

*   **Filter by service (`-u`)**:
    ```bash
    journalctl -u nginx.service
    ```
    *(Shows all logs generated by the Nginx web server).*

*   **Combine with follow (`-f`)**:
    ```bash
    journalctl -u ssh.service -f
    ```
    *(Watches live login attempts via SSH).*

---

## 3. Filtering by Time
Finding logs from a specific incident time is a very common task.

*   **Logs since a specific time**:
    ```bash
    journalctl --since "2023-10-25 08:00:00"
    ```

*   **Logs within a time window**:
    ```bash
    journalctl --since "2023-10-25 08:00:00" --until "2023-10-25 10:00:00"
    ```

*   **Using relative keywords**:
    `journalctl` understands words like `yesterday`, `today`, and relative time like `1 hour ago`.
    ```bash
    journalctl --since "1 hour ago"
    ```
    ```bash
    journalctl --since yesterday
    ```

---

## 4. Filtering by Priority (Severity)
System logs are categorized by priority (from 0 to 7).
*   **0**: emerg (System is unusable)
*   **1**: alert (Action must be taken immediately)
*   **2**: crit (Critical conditions)
*   **3**: err (Error conditions)
*   **4**: warning (Warning conditions)
*   **5**: notice (Normal but significant condition)
*   **6**: info (Informational)
*   **7**: debug (Debug-level messages)

*   **Show only Errors and above (`-p 3` or `-p err`)**:
    ```bash
    journalctl -p err
    ```
    *(Filters out all the noise and only shows actual errors, critical issues, and emergencies).*

---

## 5. Filtering by User ID (UID)
You can filter logs to only show activities related to a specific user using their User ID (UID).

*   **Find a User's UID**:
    ```bash
    id username
    ```
    *(For example, the UID for a normal user is usually `1000` or higher).*

*   **View logs for that User**:
    ```bash
    journalctl _UID=1000
    ```
    *(Shows all log entries generated by processes owned by the user with UID 1000).*

*   **Combine with follow (`-f`)**:
    ```bash
    journalctl _UID=1000 -f
    ```

---

## 6. Viewing Boot Logs
If your server crashed or rebooted unexpectedly, you need to check the boot logs.

*   **Logs from the current boot (`-b`)**:
    ```bash
    journalctl -b
    ```

*   **Logs from the previous boot**:
    ```bash
    journalctl -b -1
    ```
    *(Requires persistent logging to be enabled).*

---

## 7. Output Formatting
Sometimes you need logs in a specific format for parsing or exporting.

*   **JSON Format**:
    ```bash
    journalctl -o json-pretty
    ```
*   **Kernel Messages Only** (Similar to `dmesg`):
    ```bash
    journalctl -k
    ```
*   **No Pager** (Print straight to standard output, useful for piping to `grep`):
    ```bash
    journalctl --no-pager
    ```

---

## 8. Managing Disk Space
The journal can grow very large over time. `journalctl` provides tools to manage its size.

*   **Check how much disk space the journal is using**:
    ```bash
    journalctl --disk-usage
    ```

*   **Delete logs older than a specific time (Vacuum by time)**:
    ```bash
    sudo journalctl --vacuum-time=14d
    ```
    *(Deletes all logs older than 14 days).*

*   **Limit the journal to a specific size (Vacuum by size)**:
    ```bash
    sudo journalctl --vacuum-size=500M
    ```
    *(Deletes oldest logs until the journal size is exactly 500 MB).*

---

## 9. Practical Troubleshooting Scenarios

### Scenario A: The Web Server Crashed
**The Problem**: Nginx stopped working at around 3:00 PM today.
**The Solution**:
```bash
journalctl -u nginx.service --since "15:00:00" --until "15:30:00"
```
*(This isolates exactly what the Nginx service reported during the time window of the crash).*

### Scenario B: Finding the Cause of a System Reboot
**The Problem**: The server rebooted overnight, and you want to see if there were any kernel panics or out-of-memory (OOM) errors before it went down.
**The Solution**:
```bash
journalctl -b -1 -k -p err
```
*(This shows only the **Error** level logs (`-p err`) from the **Kernel** (`-k`) during the **Previous Boot** (`-b -1`)).*

### Scenario C: Cleaning Up a Full Disk
**The Problem**: You get a "No space left on device" error. You check `journalctl --disk-usage` and find that the logs are taking up 10 GB of space.
**The Solution**:
```bash
sudo journalctl --vacuum-size=1G
```
*(Instantly frees up 9 GB of disk space by discarding the oldest logs).*
